git

介绍

历史

集中式版本管理系统：
版本存储在中央服务器中，每次修改需要拉取到本地进行修改
分布式版本管理系统：
每个人的本地都有一个完整的代码版本，可以互相之间进行推送修改，在实际的使用过程中会有一个“中央服务器”来交换大家之间的修改
​

发展

由 Linus 为了管理 Linux 系统的开发通过 C 语言编写创建的分布式版本管理系统
​

使用

本地仓库操作

本地新建数据库：
git init
​

克隆远程的数据库：
git clone
​

将文件添加到暂存区：
git add file.txt
​

丢弃暂存区中的修改：
git checkout – …
​

将暂存区的内容添加到本地仓库：
git commit -m ‘xxx’
​

从本地仓库中删除提交的文件：
git rm …
​

提交过程中的注意事项：

  • 对数据库中的内容进行修改提交之后，会生成上次提交的状态与当前状态的差异记录（revision）
  • 提交按照时间的序列在数据库中进行排序
  • git 会为每一次的提交生成一个 40 位的由英文和数字组合的值来给提交进行命名
  • 不同类别的修改（bug 修复 or 系统升级）要分开进行提交，方便后续进行查看
  • 提交的时候需要输入提交的信息，否则无法提交，多人合作的时候要统一提交的格式

建立和远程仓库分支的连接：

更新远程分支的修改：
git pull

分支

注意事项：

  • 分支是为了将修改记录的整体流程分叉保存
  • 分叉之后的分支不受其它分支影响，可以对多个分支同时进行修改
  • 开发的过程可以基于原分支进行创建新的分支，在新分支上开发完成后再将该分支合并回原分支

​

新建分支：
git branch …
​

分支切换：
git checkout …
git switch …
​

新建分支并切换到该分支：
git checkout -b …
git switch -c …

HEAD：
HEAD 指向的是现在使用中的分支的最后一次更新。通常默认指向 master 分支的最后一次更新。通过移动 HEAD，就可以变更使用的分支。
​

相对位置：
提交时使用(tilde)和^(caret)就可以指定某个提交的相对位置。最常用的就是相对于 HEAD 的位置。HEAD 后面加上(tilde)可以指定 HEAD 之前的提交记录。合并分支会有多个根节点，您可以用^(caret) 来指定使用哪个为根节点。

删除分支：
git branch -d …
强制删除分支：
git branch -D …
​

回退到上一个版本：
git reset (HEAD~ / commit_id)
有几种参数模式：

模式名称 HEAD 位置 索引 工作树
soft 修改 不修改 不修改
mixed 修改 修改 不修改
hard 修改 修改 修改

主要的使用场合：

  • 复原修改过的索引的状态(mixed)
  • 彻底取消最近的提交(hard)
  • 只取消提交(soft)

退回到原来的某个命令：
查看命令历史：
git reflog
根据表识 id 进行回退：
git reset …

分支合并：
merge
git merge …

合并 bugfix 分支到 master 分支时，如果 master 分支的状态没有被更改过，那么这个合并是非常简单的。 bugfix 分支的历史记录包含 master 分支所有的历史记录，所以通过把 master 分支的位置移动到 bugfix 的最新分支上，Git 就会合并。这样的合并被称为 fast-forward（快进）合并。


但是，master 分支的历史记录有可能在 bugfix 分支分叉出去后有新的更新。这种情况下，要把 master 分支的修改内容和 bugfix 分支的修改内容汇合起来。

因此，合并两个修改会生成一个提交。这时，master 分支的 HEAD 会移动到该提交上。

执行合并时，如果设定了–no-ff 选项，即使在能够 fast-forward 合并的情况下也会生成新的提交并合并。
​

汇合分支上的提交，合并到同一个分支：
git merge –squash

rebase
git rebase …
发生冲突则，解决冲突后：
git rebase –continue 继续合并
git rebase –abort 放弃合并
​

如果使用 rebase 方法进行分支合并，会出现下图所显示的历史记录。现在我们来简单地讲解一下合并的流程吧。

首先，rebase bugfix 分支到 master 分支, bugfix 分支的历史记录会添加在 master 分支的后面。如图所示，历史记录成一条线，相当整洁。
这时移动提交 X 和 Y 有可能会发生冲突，所以需要修改各自的提交时发生冲突的部分。

rebase 之后，master 的 HEAD 位置不变。因此，要合并 master 分支和 bugfix 分支，即是将 master 的 HEAD 移动到 bugfix 的 HEAD 这里。

主要使用的场合：

  • 汇合主题分支的提交，然后合并提交到目标分支。

标签

  • 轻标签
    • 添加名称
  • 注解标签
    • 添加名称
    • 添加注解
    • 添加签名

一般情况下，发布标签是采用注解标签来添加注解或签名的。轻标签是为了在本地暂时使用或一次性使用。
您可以指定标签名称以退出，或 reset 在「修改提交」的讲解，还可以简单的恢复过去特定的状态。

列出所有的标签：
git tag
显示所有的标签和注解
git tag -n

在当前的 HEAD 添加轻标签：

在对应 commit_id 上添加对应的标签：

git tag -a

注解标签直接添加注释
git tag -am “this is note”

查看标签的信息（标签是按照字母的顺序进行排序）：
git show …

删除指定的标签
git tag -d …

推送标签到远程仓库：

推送本地所有的标签到远端：
git push –tags

删除远程仓库中的标签：

其它

暂时保存分支的内容
stash
stash 是临时保存文件修改内容的区域。stash 可以暂时保存工作树和索引里还没提交的修改内容，您可以事后再取出暂存的修改，应用到原先的分支或其他的分支上。
git stash
​

查看 stash 库中存储的分支内容：
git stash list
​

恢复 stash 保存的内容，但是不删除 stash 库中的内容：
git stash apply …
​

恢复 stash 保存的内容，同时删除 stash 库中的内容：
git stash pop …
​

删除 stash 库中的内容：
git stash drop …
​

提取提交：
在 cherry-pick，您可以从其他分支复制指定的提交，然后导入到现在的分支

主要使用的场合：

  • 把弄错分支的提交移动到正确的地方
  • 把其他分支的提交添加到现在的分支

​

改写提交的历史：
在 rebase 指定 i 选项，您可以改写、替换、删除或合并提交
git rebase -i HEAD~ 汇合过去的提交
实际上，在 rebase 之前的提交会以 ORIG_HEAD 之名存留。如果 rebase 之后无法复原到原先的状态，可以用 git reset –hard ORIG_HEAD 复原到 rebase 之前的状态。
主要使用的场景：

  • 在 push 之前，重新输入正确的提交注解
  • 清楚地汇合内容含义相同的提交。
  • 添加最近提交时漏掉的档案
