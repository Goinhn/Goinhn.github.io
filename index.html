定义

spring 是一个开源框架，2003 年兴起的一个轻量级的 Java 开发框架
为了解决企业级应用开发的复杂性而创建的，简化开发
​

spring 简化 Java 开发：

  • 基于 pojo 的轻量级和最小入侵性编程
  • 通过 IOC，依赖注入（DI）和面向接口实现松耦合
  • 基于切面（AOP）和惯例进行声明式编程
  • 通过切面和模板减少样式代代码

​

springboot：
约定大于配置
优点：

  • 为所有 Spring 开发者更快的入门
  • 开箱即用，提供各种默认的配置来简化项目的配置
  • 内嵌容器简化 Web 项目
  • 没有冗余的代码生成和 XML 配置要求

​

微服务

应用由一系列小服务组合而成
微服务架构的历史
将每一个功能元素独立出来，把独立出来的功能元素进行动态组合

  • 节省了调度资源
  • 每个功能元素的服务都是一个可替换的、可独立升级的软件代码

api安全保证

开放接口存在的问题：

  • 请求来源是否合法
  • 请求参数是否被篡改
  • 请求的唯一性

​

参数签名

设置发送参数中对应的安全参数，signature、nonce、timestamp

signature = sign_function(nonce + secert + timestamp + key1value1key2value2key3value3)
将其它的参数按照字母的顺序进行排序进行拼接，同时加上最后的随机数、时间戳、后台的秘密数字。计算签名之后传递给后端进行校验，后端设置统一的拦截器进行校验。通过统一相同的签名算法。
​

timestamp 保证重放攻击，当前的链接在一定的时间内有效。
nonuc 保证当前的签名算法隐秘性，使得被找到规律性的可能降低
secert 保证加密签名的唯一性

project_management

工程能力的基本构成：

  • 代码
  • 文档
  • 项目管理

​

项目

在一定的约束条件下（限定时间、限定资源），具有明确目标的一次性任务
要素：

  • 时间（time）
  • 范围（Scope）
  • 成本（Cost）
  • 质量（Quality）质量是最不能妥协因素之一

​

软件项目管理

软件项目管理是为了使软件项目能够按照预定的成本、进度、质量顺利完成，而对人员、产品、过程和项目进行分析和管理的活动
项目管理使组织能够有效而且高效地开展项目
将开发人员的个人能力转换为企业的开发能力
​

过程

  • 项目启动和规划
    • 定义项目目标
    • 组件项目团队
    • 规划项目的执行计划
    • 启动项目的研发
  • 项目执行和监控
    • 推动项目按照既定的计划执行
    • 监控项目进展，识别项目风险
    • 必要时采取纠正措施
  • 项目总结和回顾
    • 对执行结果、项目收益和经验教训进行总结和回顾

​

项目启动和规划

常见问题：

  • 项目参与人的角色不明确
  • 对项目目标和时间计划缺乏明确定义
  • 对于项目资源缺乏准确估计

​

项目的基础设施：

  • wiki
  • hi 群
  • Email 列表
  • iCode 路径
  • iCafe 空间
  • 项目规范

​

项目执行和监控

常见问题：

  • 没有清晰的项目周报
  • 不会使用迭代管理

​

项目周报：

  • 项目的总体计划和对应的情况
  • 上周计划、本周进展、下周计划
  • 项目的风险和处置

​

项目总结和回顾

常见问题：

  • 对项目缺乏回顾总结
  • 缺乏对项目执行问题的分析

​

项目管理意识

  • 计划意识
  • 全局意识
  • 风险意识
  • 协作意识

R&D Documentation

研发文档
​

软件工程师的基本要求：

  • 代码
  • 文档
  • 项目管理

目的

  • 提高沟通的效率
  • 提升对“思考过程”的管理

​

认知

  • 文档本身也是产出
  • 修改文档的代价比修改代码的代价小
  • 项目延续时间和复杂性往往会超出预期
  • 文档时设计过程中使用的工具，和设计过程的结果
  • 文档要进行 review

​

常见文档类型

  • 需求设计文档：需求、重点、取舍过程
  • 接口文档：函数、参数、返回值
  • 关键性的算法文档：思路、关键点
  • 系统总体框架：全局的思路

​

文档结构

  • 标题
    • 标题是否表达文档内容
    • 标题是否和文档内容相符
    • 各层标题所构成的提纲是否反映文档的内容
  • 段落
    • 段首一定要有缩进
    • 段落不要太长
    • 注意每一段的第一句话
    • 每一段内的多句话一定要有逻辑
    • 多使用项目符号和编号

​

写作流程

  • 拉提纲，自顶向下
  • 做好 review

zsh

zsh：shell 工具
oh-my-zsh：zsh 搭配的开源配置的工具
​

安装 oh-my-zsh

所有的配置都在.zshrc 文件中

安装缺少的字体

在 iterm 中选择安装的字体

配置自动提醒插件

​

zsh 环境变量文件：

  • .zshenv
    • .zshenv 总是被读取,所以通常把$PATH,$EDITOR 等变量写在这里,这样无论是在 shell 交互,或者运行程序都会读取此文件
  • .zshrc
    • .zshrc 主要用在交互 shell,所以主要是为 shell 服务的,比如对 shell 做的一些个性化设置都可以在这里写入
  • .zlogin
    • .zlogin 在 login shell 的时候读取,所以比如 X 系统启动的时候会读取此文件,所以不会再运行中重复读取
  • .zprofile
    • .zprofile 是给 ksh 用户的一个.zlogin 的替代品,所以我们如果使用了.zlogin 就不必再关心此文件
  • .zlogout
    • .zlogout 这个就很好理解了, 退出终端的时候读取,用于做一些清理工作,一般我们也用不上

读取的顺序：
.zshenv → [.zprofile if login] → [.zshrc if interactive] → [.zlogin if login] → [.zlogout sometimes].
