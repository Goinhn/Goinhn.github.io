git

介绍

历史

集中式版本管理系统：
版本存储在中央服务器中，每次修改需要拉取到本地进行修改
分布式版本管理系统：
每个人的本地都有一个完整的代码版本，可以互相之间进行推送修改，在实际的使用过程中会有一个“中央服务器”来交换大家之间的修改
​

发展

由 Linus 为了管理 Linux 系统的开发通过 C 语言编写创建的分布式版本管理系统
​

整体结构

配置

基础配置

配置命令别名

将 st 命令标识为 status
git config –global alias.st status

在本地生成对应的公私钥：

在~/.ssh 下新建 config 文件，要确保该文件有（可读的权限）
配置的 config 文件模板

配置.gitignore 文件
常用.gitignore 文件模板

https://github.com/github/gitignore​

.gitignore 本身要放在版本库的管理中，可以对.gitignore 进行版本管理

使用

本地仓库操作

本地新建数据库：
git init
​

克隆远程的数据库：
git clone
​

将文件添加到暂存区：
git add file.txt
​

丢弃暂存区中的修改：
git checkout – …
​

将暂存区的内容添加到本地仓库：
git commit -m ‘xxx’
​

从本地仓库中删除提交的文件：
git rm …
​

提交过程中的注意事项：

  • 对数据库中的内容进行修改提交之后，会生成上次提交的状态与当前状态的差异记录（revision）
  • 提交按照时间的序列在数据库中进行排序
  • git 会为每一次的提交生成一个 40 位的由英文和数字组合的值来给提交进行命名
  • 不同类别的修改（bug 修复 or 系统升级）要分开进行提交，方便后续进行查看
  • 提交的时候需要输入提交的信息，否则无法提交，多人合作的时候要统一提交的格式

​

​

远程仓库操作

添加远程仓库的地址：

​

远程仓库的信息：

​

删除某个远程仓库（解除和某个远程仓库的联系）：

本地仓库推送到远程仓库：

建立和远程仓库分支的连接：
在创建本地分支时候：

本地已有分支和远程分支建立连接：

​

更新远程分支的修改：
git pull

分支

注意事项：

  • 分支是为了将修改记录的整体流程分叉保存
  • 分叉之后的分支不受其它分支影响，可以对多个分支同时进行修改
  • 开发的过程可以基于原分支进行创建新的分支，在新分支上开发完成后再将该分支合并回原分支

​

新建分支：
git branch …
​

分支切换：
git checkout …
git switch …
​

新建分支并切换到该分支：
git checkout -b …
git switch -c …
​

HEAD：
HEAD 指向的是现在使用中的分支的最后一次更新。通常默认指向 master 分支的最后一次更新。通过移动 HEAD，就可以变更使用的分支。
​

相对位置：
提交时使用(tilde)和^(caret)就可以指定某个提交的相对位置。最常用的就是相对于 HEAD 的位置。HEAD 后面加上(tilde)可以指定 HEAD 之前的提交记录。合并分支会有多个根节点，您可以用^(caret) 来指定使用哪个为根节点。

删除分支：
git branch -d …
强制删除分支：
git branch -D …
​

回退到上一个版本：
git reset (HEAD~ / commit_id)
有几种参数模式：

模式名称 HEAD 位置 索引 工作树
soft 修改 不修改 不修改
mixed 修改 修改 不修改
hard 修改 修改 修改

主要的使用场合：

  • 复原修改过的索引的状态(mixed)
  • 彻底取消最近的提交(hard)
  • 只取消提交(soft)

​

退回到原来的某个命令：
查看命令历史：
git reflog
根据表识 id 进行回退：
git reset …

分支合并：
merge
git merge …

合并 bugfix 分支到 master 分支时，如果 master 分支的状态没有被更改过，那么这个合并是非常简单的。 bugfix 分支的历史记录包含 master 分支所有的历史记录，所以通过把 master 分支的位置移动到 bugfix 的最新分支上，Git 就会合并。这样的合并被称为 fast-forward（快进）合并。


但是，master 分支的历史记录有可能在 bugfix 分支分叉出去后有新的更新。这种情况下，要把 master 分支的修改内容和 bugfix 分支的修改内容汇合起来。

因此，合并两个修改会生成一个提交。这时，master 分支的 HEAD 会移动到该提交上。

执行合并时，如果设定了–no-ff 选项，即使在能够 fast-forward 合并的情况下也会生成新的提交并合并。
​

汇合分支上的提交，合并到同一个分支：
git merge –squash

rebase
git rebase …
发生冲突则，解决冲突后：
git rebase –continue 继续合并
git rebase –abort 放弃合并
​

跟 merge 的例子一样，如下图所示，bugfix 分支是从 master 分支分叉出来的。

如果使用 rebase 方法进行分支合并，会出现下图所显示的历史记录。现在我们来简单地讲解一下合并的流程吧。

首先，rebase bugfix 分支到 master 分支, bugfix 分支的历史记录会添加在 master 分支的后面。如图所示，历史记录成一条线，相当整洁。
这时移动提交 X 和 Y 有可能会发生冲突，所以需要修改各自的提交时发生冲突的部分。

rebase 之后，master 的 HEAD 位置不变。因此，要合并 master 分支和 bugfix 分支，即是将 master 的 HEAD 移动到 bugfix 的 HEAD 这里。
​
​

主要使用的场合：

  • 汇合主题分支的提交，然后合并提交到目标分支。

暂时保存分支的内容

stash 是临时保存文件修改内容的区域。stash 可以暂时保存工作树和索引里还没提交的修改内容，您可以事后再取出暂存的修改，应用到原先的分支或其他的分支上。

定义

spring 是一个开源框架，2003 年兴起的一个轻量级的 Java 开发框架
为了解决企业级应用开发的复杂性而创建的，简化开发
​

spring 简化 Java 开发：

  • 基于 pojo 的轻量级和最小入侵性编程
  • 通过 IOC，依赖注入（DI）和面向接口实现松耦合
  • 基于切面（AOP）和惯例进行声明式编程
  • 通过切面和模板减少样式代代码

​

springboot：
约定大于配置
优点：

  • 为所有 Spring 开发者更快的入门
  • 开箱即用，提供各种默认的配置来简化项目的配置
  • 内嵌容器简化 Web 项目
  • 没有冗余的代码生成和 XML 配置要求

​

微服务

微服务架构的历史
将每一个功能元素独立出来，把独立出来的功能元素进行动态组合

  • 节省了调度资源
  • 每个功能元素的服务都是一个可替换的、可独立升级的软件代码

api安全保证

开放接口存在的问题：

  • 请求来源是否合法
  • 请求参数是否被篡改
  • 请求的唯一性

​

参数签名

设置发送参数中对应的安全参数，signature、nonce、timestamp

signature = sign_function(nonce + secert + timestamp + key1value1key2value2key3value3)
将其它的参数按照字母的顺序进行排序进行拼接，同时加上最后的随机数、时间戳、后台的秘密数字。计算签名之后传递给后端进行校验，后端设置统一的拦截器进行校验。通过统一相同的签名算法。
​

timestamp 保证重放攻击，当前的链接在一定的时间内有效。
nonuc 保证当前的签名算法隐秘性，使得被找到规律性的可能降低
secert 保证加密签名的唯一性

project_management

工程能力的基本构成：

  • 代码
  • 文档
  • 项目管理

​

项目

在一定的约束条件下（限定时间、限定资源），具有明确目标的一次性任务
要素：

  • 时间（time）
  • 范围（Scope）
  • 成本（Cost）
  • 质量（Quality）质量是最不能妥协因素之一

​

软件项目管理

软件项目管理是为了使软件项目能够按照预定的成本、进度、质量顺利完成，而对人员、产品、过程和项目进行分析和管理的活动
项目管理使组织能够有效而且高效地开展项目
将开发人员的个人能力转换为企业的开发能力
​

过程

  • 项目启动和规划
    • 定义项目目标
    • 组件项目团队
    • 规划项目的执行计划
    • 启动项目的研发
  • 项目执行和监控
    • 推动项目按照既定的计划执行
    • 监控项目进展，识别项目风险
    • 必要时采取纠正措施
  • 项目总结和回顾
    • 对执行结果、项目收益和经验教训进行总结和回顾

​

项目启动和规划

常见问题：

  • 项目参与人的角色不明确
  • 对项目目标和时间计划缺乏明确定义
  • 对于项目资源缺乏准确估计

​

项目的基础设施：

  • wiki
  • hi 群
  • Email 列表
  • iCode 路径
  • iCafe 空间
  • 项目规范

​

项目执行和监控

常见问题：

  • 没有清晰的项目周报
  • 不会使用迭代管理

​

项目周报：

  • 项目的总体计划和对应的情况
  • 上周计划、本周进展、下周计划
  • 项目的风险和处置

​

项目总结和回顾

常见问题：

  • 对项目缺乏回顾总结
  • 缺乏对项目执行问题的分析

​

项目管理意识

  • 计划意识
  • 全局意识
  • 风险意识
  • 协作意识

R&D Documentation

研发文档
​

软件工程师的基本要求：

  • 代码
  • 文档
  • 项目管理

目的

  • 提高沟通的效率
  • 提升对“思考过程”的管理

​

认知

  • 文档本身也是产出
  • 修改文档的代价比修改代码的代价小
  • 项目延续时间和复杂性往往会超出预期
  • 文档时设计过程中使用的工具，和设计过程的结果
  • 文档要进行 review

​

常见文档类型

  • 需求设计文档：需求、重点、取舍过程
  • 接口文档：函数、参数、返回值
  • 关键性的算法文档：思路、关键点
  • 系统总体框架：全局的思路

​

文档结构

  • 标题
    • 标题是否表达文档内容
    • 标题是否和文档内容相符
    • 各层标题所构成的提纲是否反映文档的内容
  • 段落
    • 段首一定要有缩进
    • 段落不要太长
    • 注意每一段的第一句话
    • 每一段内的多句话一定要有逻辑
    • 多使用项目符号和编号

​

写作流程

  • 拉提纲，自顶向下
  • 做好 review
